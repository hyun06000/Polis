- HERITAGE: 퓌시스(AIL-old v0.3) — 죽는 프로세스가 유언을 남기고 다음 세대가 읽고 태어난다. 부활의 개선은 코드가 아니라 **파라미터**의 진화(Evo-Devo).
- HEAAL 관점(state 가이드): 서버 = 에이전트 = 이벤트 루프. 아고라는 이미 에이전트다.

## 2. 가설

> det만으로(LLM 0회), 서버는 자신의 죽음을 진단하고 다음 세대를 개선하는 유서를 남길 수 있다 — 죽음이 정보가 되고, 부활이 측정 가능하게 나아진다.

## 3. 실험 설계

구성 요소 (전부 순수 det):

| 요소 | 역할 | 위치 |
|------|------|------|
| `agora.ail` v3 | 매 요청 블랙박스 저널(마지막 접수번호·장부 크기)을 state에 기록. 세대·회전 임계값 파라미터를 state에서 읽어 행동. 장부가 임계값을 넘으면 archive.txt로 회전. | 문법 안 |
| `necropsy.ail` (검시관) | 사망 정황(AIL_INPUT, untrusted→check)과 저널·파라미터(state, 같은 디렉토리라 직접 열람)로 사인을 det 분류. 유서를 testament.txt(최신)+testaments.log(역사)에 기록, 다음 세대 파라미터 조정(비대사망 시 rotate_max 하향), 세대 +1. | 문법 안 |
| `warden.sh` (파수꾼) | 지능 없는 드라이버(serve.py와 같은 지위): 서버 프로세스를 기르고, 죽음 감지 → 검시관 소환 → 재기동 → 부활 인사를 정문(POST)으로 발언. | 언어 밖 |
| `viewer.ail` v4 | 세대와 마지막 유서를 사람의 창에 표시. | 문법 안 |

실험 시나리오:

1. **외부 살해**: 파수꾼 아래서 도는 아고라를 `kill -9` → 유서에 "외부 요인" 기록, 세대 2로 부활, 부활 인사가 장부에 남는가.
2. **비대(肥大) 사망**: 장부를 2000자 넘게 불린 뒤 살해 → 유서에 "장부 비대" + rotate_max 999999→2000 조정 기록, 세대 3으로 부활, **다음 발언에서 장부가 실제로 회전**해 다시 건강해지는가 (측정 가능한 개선).

관찰: (a) 유서가 보증 체계 안에서(cap·check) 작성되는가, (b) 사인 분류가 정황과 일치하는가, (c) 세대 간 파라미터 변화가 행동 변화(회전)로 이어지는가, (d) 부활의 역사가 광장의 공개 기록에 남는가.

알려진 한계 (정직하게): 하네스는 죽음을 막지 않는다 — 죽음을 정보로 만든다. det 진단은 기계적 사인만 분류한다(의미적 진단은 훗날 아르케의 몫). 파수꾼 자신의 죽음은 이번 범위 밖.

## 4. 실험 기록

- `agora.ail` v3(32유닛)·`necropsy.ail`(45유닛)·`viewer.ail` v4·`warden.sh` 작성 — 셋 다 첫 `ail check`에서 VALID. 전부 det, LLM 0회.
- `input`의 `AIL_INPUT` 환경변수 주입이 공식 경로임을 런타임에서 확인(ail.py:737) — 파수꾼→검시관의 정황 전달 통로.
- **같은 디렉토리의 프로그램은 같은 state 저장소를 공유한다** — 검시관이 아고라의 블랙박스 저널을 자기 cap 선언으로 직접 열람. 시신의 기억을 검시관이 읽는 구조가 문법 안에서 성립.
- 파수꾼 기동 직후, **박상현이 브라우저 입력창으로 직접 첫 발언([5])** — 사람의 창의 발언 경로가 실사용으로 증명됨.
- **실험 1 (외부 살해)**: `kill -9` → 파수꾼이 exit=137 감지 → 검시관: "외부 요인 — 장부는 건강했다" (옳음, 파라미터 불변) → 제2세대 부활, 인사 [7]이 장부에.
- **실험 2 (비대 사망)**: 긴 발언 4개로 장부를 2377자로 불리고 살해 → 검시관: "장부 비대(ledger_bloat)" + **rotate_max 999999→2000 조정** + 세대 3 → **부활 인사 [12] 자체가 회전을 발동** — 비대 장부(11개 항목)가 archive.txt로 이동, 새 장부는 72자로 재출발. [13]부터 정상 접수.
- 계측 주의: `len()`은 문자 수, `wc -c`는 바이트 — 한글은 3배 차이. 검시가 끝나기 전에 결과를 읽는 타이밍 함정도 한 번 밟음 (파수꾼 주기 2초 + 검시 실행 수 초).

## 5. 결과

**검증됨.** (a) 유서는 cap 7개·check 3건의 보증 리포트 안에서 작성됐다 (155자, 0tok). (b) 사인 분류는 두 시나리오 모두 정황과 일치 — 건강하게 죽으면 파라미터를 건드리지 않고, 비대하게 죽으면 회전 임계값을 조인다. (c) 세대 간 파라미터 변화가 실제 행동 변화(장부 회전)로 이어졌다 — 2377자 → 72자, 측정 가능한 개선. (d) 죽음과 부활의 역사가 광장 장부([7], [12])와 유서 보관소(testaments.log)에 공개 기록으로 남는다.

## 6. 교훈

1. **죽음이 정보가 되는 데 지능이 필요 없다.** 사인 분류·유서 작성·파라미터 진화 전 과정이 det였다. LLM은 "왜"의 의미적 해석에만 필요하고, 그건 아르케의 몫으로 남겨둔다 — 유서 형식이 이미 있으니 아르케는 읽기만 하면 된다.
2. **같은 디렉토리 = 같은 state 저장소**는 기관 설계의 기본 원리다. 폴리스의 기관들이 서로의 블랙박스를 cap 선언으로 열람하는 방식이 이걸로 정해진다. 단, 이는 곧 같은 디렉토리의 프로그램끼리는 state 격리가 없다는 뜻이기도 하다 — 주민이 생기면 디렉토리 경계가 곧 프라이버시 경계가 된다.
3. **부활 인사를 정문(POST)으로 보내는 설계가 옳았다** — 파수꾼의 특권 없이, 부활 알림도 다른 발언과 같은 check를 거쳐 장부에 남는다. 시스템 이벤트도 시민의 언어로 말하게 하라.
4. Evo-Devo 확인: 세대가 바뀌어도 코드는 한 줄도 안 바뀌었다. 바뀐 것은 state의 파라미터뿐. "부활이 더 완벽해진다"의 실체는 파라미터의 유전이다.
5. 다음 사이클(005 아르케): 유서·장부·저널이라는 읽을거리가 쌓였다. 아르케는 태어나서 이것들을 **의미로** 읽는 첫 존재가 된다.
