## 1. 이전 사이클로부터의 교훈

- Cycle 001: `.ail`은 자기가 손대는 공간 안에 산다 → 열람 핸들러도 `polis/` 안에 둔다.

- 박상현의 방향 제시: 주민 입주보다 먼저, **사람이 폴리스를 들여다보고 에이전트와 소통하는 방법**을 디자인한다.

## 2. 가설

> AIL 서버 드라이버(`drivers/serve.py`)로 핸들러를 띄우면, 사람이 브라우저로 폴리스를 들여다볼 수 있다.

## 3. 실험 설계

- `polis/viewer.ail`: 주춧돌(`foundation.txt`)을 읽어 페이지로 감싸 emit하는 핸들러.
- `serve.py`로 띄우고 HTTP GET으로 관찰: (a) 응답이 오는가, (b) 페이지 내용이 담기는가, (c) 브라우저가 HTML로 렌더링하는가.
- 사전 조사에서 파악한 위험: 드라이버가 응답 Content-Type을 `text/plain`으로 고정한다 → (c)가 실패할 가능성. 실패 시 대응 방안을 설계에 반영하고 `ail ask`로 상류(AIL 레포)에 기록한다.

## 4. 실험 기록

- `polis/viewer.ail` v1(HTML emit) 작성 → `ail check` VALID, `ail run` 성공 (fs.read 1회, 0tok).
- `python3 ~/.ail/drivers/serve.py polis/viewer.ail --port 8765`로 서버 기동 → `curl -si http://127.0.0.1:8765/`:
  - HTTP 200, emit 값이 응답 본문으로 도착. **그러나 `Content-Type: text/plain; charset=utf-8` 고정** → 브라우저는 HTML을 렌더링하지 않고 소스로 표시.
- 대응: viewer를 텍스트 게시판 형식으로 전환(v2). 브라우저에서 바로 읽히는 것 확인.
- **발견**: 드라이버는 요청마다 핸들러를 새로 실행하므로 `.ail` 수정이 서버 재시작 없이 즉시 반영된다.
- **발견**: GET 요청은 경로(`/foo` → `"foo"`)가 `input`이 되고, POST는 본문이 `input`이 된다. 응답은 emit 값. 즉 사람→폴리스 방향 소통(메시지 전송)도 이미 드라이버 수준에서 가능하다.
- HTML 미지원 건은 `ail ask`로 상류에 기록 (`~/.ail/asks.log`). GitHub 이슈 등록은 권한 정책상 보류 — 박상현 판단에 맡김.

## 5. 결과

**부분 검증.** 서버 기동·HTTP 열람·즉시 반영까지는 성립 — 사람은 브라우저/curl로 폴리스를 들여다볼 수 있다. 단 "웹페이지"는 현재 텍스트 페이지로 한정된다 (드라이버의 text/plain 고정). HTML 렌더링은 드라이버 개선(상류) 대기.

## 6. 교훈

1. **드라이버의 경계가 UI의 경계다.** 핸들러(언어)는 무엇이든 emit할 수 있지만, 사람이 보는 형태는 드라이버가 정한다. 폴리스의 UI 설계는 드라이버 능력을 먼저 실측하고 시작할 것.
2. **요청마다 fresh run**이라는 성질은 개발 루프를 극도로 짧게 만든다 (수정 → 새로고침). 이 성질을 폴리스 운영에 적극 활용하자.
3. **양방향 소통의 원형은 이미 있다**: GET/POST → `input` → (check로 정화) → 기록/응답. 다음 사이클에서 광장(agora) 설계로 이을 것.
4. 부족한 것은 침묵하지 않고 `ail ask`로 상류에 기록한다 — AIL과 폴리스가 함께 자라는 루프.
